iOS应用架构谈 view层的组织和调用方案

发表于 2018-05-01 | 分类于 斑马日报 2018/05/01 第1期 | | 阅读次数:

适合: 高级开发者阅读
来自: Casa Taloyum
关键字: 架构
原文：iOS应用架构谈 view层的组织和调用方案

Casa Taloyum的iOS应用架构谈系列第二篇，也是最重要的一篇。View层和业务的对接面最广，影响也最深。因此如果View层的架构不够好，哪怕稍微一点点的改动而引起的业务方的问题，会如同蝴蝶效应一样，让你进退两难。所以，诚如Casa Taloyum在文章中说的，在做View层架构的时候，我们必须抱着充分的自我怀疑态度，做决策时要拿捏好尺度。

此外，你要要知道，我们熟悉或不熟悉的MVC、MVVM、MVP、VIPER等架构模式，大部分就是针对View层而设计的，随便拿一个出来就可以说上好几篇幅。

Casa Taloyum在这篇文中大致讨论了下面的这些问题：

  • View代码结构的规定
  • 关于view的布局
  • 何时使用storyboard，何时使用nib，何时使用代码写View
  • 是否有必要让业务方统一派生ViewController？
  • MVC、MVVM、MVCS、VIPER
  • 跨业务时View的处理

同样，在您阅读之前，可以考虑下这些问题，欢迎在评论区留言。

原文：iOS应用架构谈 本地持久化方案及动态部署

Casa Taloyum的iOS应用架构谈系列第四篇。这一篇的持久化方案又是一个非常需要发挥的地方，方案非常多，非常值得讨论。持久化方案对整个App架构的影响和网络层方案对整个架构的影响类似，一般都是导致整个项目耦合度高的罪魁祸首。

所以这篇文章着重讨论了

  • 根据需求决定持久化方案
  • 持久层与业务层之间的隔离
  • 持久层与业务层的交互方式
  • 数据迁移方案
  • 数据同步方案

我在做项目的时候，会非常纠结解耦和同步问题，所以会经常回过头来查阅这篇文章给的方案。所以如果你有更好的方案或者想法，欢迎留言讨论。

至于iOS动态部署方案，事实上也可以另开一篇的内容，不过Casa Taloyum把他们和在一起了。随着Apple对动态部署方案的审核愈加严格，所以请更加慎重的选择方案。

在发布这篇文章的时候，Casa Taloyum也发布了CTPersistance和CTJSBridge这两个库，因此也欢迎大家下载使用或者分析。
